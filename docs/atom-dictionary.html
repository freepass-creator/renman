원자 사전 279

데이터를 숫자·텍스트가 아니라 뜻으로. 우리가 모으는 모든 원자가 무엇이고·왜 모으고·무엇을 구동하는지. 저장은 사실 원자만, 미수·분개·상태·집계 같은 파생은 엔진이 계산한다.

0층 · 사실 원자
사람·차·돈·서류가 남긴 원본값. 계약일·대여료·plate·txDate·ownerRegNo. OCR·엑셀·자금일보에서 수집, 저장만 되고 계산 안 됨.
1층 · 분자 ←엔진
원자를 엮어 만든 파생 구조체. generateSchedules→미수, buildMatchIndex→수납매칭, buildBankJournal→분개, simpleVehicleState→상태.
2층 · 업무흐름
분자가 생업을 돌린다. 소싱→상품화→계약→인도→수납→미수→회수→반납정산→매각. 전이는 원자·분자가 자동 게이팅.
검색 결과 없음

핵심 원자 사슬

원자가 홀로 살지 않고 엔진에 먹혀 업무를 만드는 경로. 이 사슬대로 데이터를 넣으면 스케줄·미수·경보로 살아난다.

  1. 수납 생성 사슬: 월대여료+계약기간(=회차수)+결제일+선후불 → generateSchedules → 회차 스케줄(dueDate·amount) → recalcContract가 반납일 cutoff 반영 → 미수(unpaidAmount)·미납회차수·현재회차 파생 → RiskIssue '미납'/시동제어/회수 SLA(overdueDays) → 반납 시 잔액이면 endReason '채권보전'·status '채권'
  2. 수납 매칭·미수차감 사슬: 은행거래(txDate·amount·counterparty·contractNo) → buildMatchIndex(계약번호 정확키>이름+별칭>차번끝4>금액) → PaymentEntry(txId·source) 생성 → 회차 status '완료' → unpaidAmount 자동 차감 → 감사로그(자동매칭 라벨). 해제 시 txId 역참조로 원복
  3. 회사 귀속·코드 발급 사슬: 자동차등록증 ownerRegNo → findCompanyByRegNo(사업자/법인번호 정규화 비교) → company 귀속 → 회사코드(CP01) prefix → nextAssetCode(CP02VH0001)·nextContractNo(CP01-2607-0001) → v6 문서ID ${companyId}__${자연키} 1:1 이관
  4. 차량 정합 축 사슬: plate → normPlate(O→0·I→1) → vehicleMatchesPlate(+plateHistory 번호변경 이력) → 계약↔차량↔보험↔과태료 단일 축 연결 → upsertVehicleFromContract가 없는 차량 자동 자산생성·currentContractId 세팅 → findVehicleForContract(vehicleId 우선, plate 폴백)
  5. 자산 감가·처분 사슬: 매입가(purchasePrice, 없으면 매매 contractDocPrice·임대차 가드) + 취득일(purchasedDate>acquisitionDate>최초등록일) → 정액 감가(잔존10%/60개월)·감가누계·장부가 → 매각가−장부가=처분손익 → 결산분개(감가상각비/감가상각누계액, 처분이익408/처분손실518)
  6. 현금→회계 변환 사슬: subject(계정과목) → RECEIPT_TO_ACCOUNT/EXPENSE_TO_ACCOUNT → JournalEntry 차/대변 → 계정잔액(normalSide) → 손익. 보증금수령→DEPOSIT_LIAB(부채), 차량매입→VEHICLE(자산), 자금이동(INTERNAL)→GL 완전 제외로 이중계상 차단
  7. CMS 집금 풀이 사슬: 효성CMS 뭉텅이 입금(settlementId deposit) ↔ 개별 CMS/카드(item) → SettlementMatch(gross·net·fee=수수료 2~3%) → deposit만 대여료수입(gross)+지급수수료(fee) 2분개, item은 분개·매칭 제외 → CmsRow.plate(회원명 정규식)로 계약별 배분
  8. 과태료 실운전자 구상 사슬: 고지서 위반일시×plate → matchDriver(계약 start≤vdate≤end) → 실운전자 특정 → 변경부과 상태(접수→변경부과완료) → 부과 청구처(계약 endReason 기반: 정상=계약자, 중도해지·채권보전=회사 부담)
  9. 차량 상태 양방향 동기 사슬: 차량 세부 status(19단계, SoT) → PREP_STAGE(준비축 6값)×SaleAxis(판매축 4값) = simpleVehicleState 헤드라인 → 운영현황 노출/가동률 필터(INACTIVE/IDLE/DISPOSAL 셋) ↔ syncContractStatusFromVehicle로 활성계약 vehicleStatus 역전파
  10. 신뢰·재현 레이어 사슬: 증명서 OCR 파싱 → _ocrOriginal 원본 스냅샷 영구보존 → 정규화 필드 → 자연키 dedup(_key=회사__plate/contractNo/복합키)로 재업로드 멱등 → 감사로그(action·entityType·before/after·actor) → soft-delete(deletedAt, 물리삭제 금지)

이걸로 뭘 하나 — v6가 "도메인 원자"로 데이터를 다루려면

제네릭 EntityRecord(숫자/텍스트 칸) 위에 원자의 의미·관계·구동력을 얹는 6가지.

v6(jpkerp6-app)는 이미 제네릭 EntityRecord 파이프라인을 갖췄다: lib/intake/entities.ts의 ENTITIES 스키마(FieldDef+idFrom/keyFields), lib/store.ts의 naturalKey→`${companyId}__${key}` 문서ID·dedup, lib/tenant.ts 회사격리. 문제는 이 구조가 필드를 여전히 '타입 있는 칸(number/text/date)'으로만 다뤄, 원자가 품은 의미(왜 모으나)·관계(무엇과 연결)·구동력(무엇을 만드나)이 스키마 밖 사람 머릿속에만 있다는 점이다. 데이터를 도메인 원자로 다루려면 6가지를 EntityRecord 위에 얹어야 한다.

1
원자 메타를 스키마 1급 시민으로

각 FieldDef에 이 사전이 정리한 (whyCollect·drives·connects·source·derivedBy)를 선언적으로 부착한다. jpkerp5 lib/types.ts의 풍부한 주석 지식(강타입 도메인)을 v6 스키마의 구조화 메타로 승격하면, 필드가 '무엇을 구동하고 무엇과 연결되는지'가 코드에서 조회 가능해져 고아 입력(캡처만 하고 안 쓰는 필드) 감사가 자동화된다.

2
접착제 원자를 '값'이 아닌 '관계 해석기'로 등록

normPlate·vehicleId·contractNo·ownerRegNo·companyId·settlementId를 단순 문자열 필드가 아니라 resolver로 규정한다. 저장 시 정규화·자연키·FK 확정(findVehicleForContract 패턴)이 자동 실행돼 계약↔차량↔보험↔과태료↔자금이 물리 축으로 묶인다. plate 정규화 오차(O/I)·번호변경(plateHistory)을 여기서 흡수한다.

3
파생 엔진을 원자 위 순수함수로 이식

v5의 recalcContract·buildMatchIndex·computeAssetLedgerEntry·buildBankJournal·simpleVehicleState를 v6에 SSOT로 옮긴다(이미 contract-ops·risk-ops·cash-ledger·operating-snapshot·classify-subject로 부분 이식됨). 저장은 사실 원자만, 분자(미수·분개·상태·집계)는 read transform으로 매번 재산출 — DB에 파생값을 굳히지 않는다.

4
도메인 상수를 코드 단일출처로

LEDGER_SUBJECTS(계정과목)·차량 19상태·상태전이 게이트·감가정책·리스크 12종을 상수 SSOT로 두고, 페이지 인라인 판단(c.status==='반납' 류)을 금지한다(contract-lifecycle.ts가 25곳 파편화를 통합한 교훈). v5 40여 세분 계정을 v6 19종 티어형으로 압축하되 JBO_SUBJECT_MAP 같은 흡수 매핑으로 원문 무손실.

5
신뢰 레이어를 EntityRecord 베이스에 내장

_ocrOriginal 원본 스냅샷·자연키 dedup·감사로그(before/after·actor)·soft-delete·낙관적잠금(updatedAt)·companyId 격리를 모든 엔티티가 상속하는 공통 베이스로 만든다. 이러면 원자 하나가 바뀌어도 누가·언제·무엇을·원본대비를 재구성할 수 있다.

6
티어형 응집

베이스 티어(운영+리스크)는 최소 원자셋(plate·계약자·기간·대여료·상태·미수)만으로 자급자족 응집하고, 심화 원자(자산 감가·복식부기 GL·정산 대사)는 opt-in으로 얹는다. 핵심은 '숫자를 저장'하는 게 아니라 '뜻을 저장하고 파생은 엔진이 계산'하는 전환이다.

응집 서사

렌터카 ERP가 응집하는 이유는 279개 원자가 "숫자/텍스트"가 아니라 서로를 물고(관계) → 엔진에 먹혀 파생 분자를 만들고(계산) → 그 분자가 업무 판단을 구동하는(경보·분개·회수단계) 3층 구조로 짜여 있기 때문이다. [0층·사실 원자] 사람·차·돈·서류가 남긴 원본값. 계약일·월대여료·결제일·선후불(계약), plate·vin·취득가(차량), txDate·amount·counterparty·subject(자금), ownerRegNo·회사코드(회사). 대부분 OCR 증명서·엑셀·자금일보에서 들어오며 각자 "왜 모으나(업무목적)"와 "어디에 연결되나"를 이미 품고 있다. 이 원자들은 저장만 되고 계산되지 않는다(SSOT의 '사실3'). [1층·분자 — 엔진이 원자를 엮어 만든 파생 구조체] 개별 원자는 홀로 살지 않고 순수함수 엔진에 들어가 의미 있는 복합체가 된다. ①generateSchedules가 (계약일+결제일+선후불+기간+월대여료)를 물어 회차 스케줄 분자(dueDate·amount 박힌 배열)를 만들고, recalcContract가 여기서 미수(unpaidAmount)·현재회차·연체상태를 뽑는다(payment-schedule.ts:93-138 실측). ②buildMatchIndex가 (계약번호>계약자명+별칭+주운전자>차번끝4>금액)을 색인해 입금 한 줄을 어느 계약 회차로 귀속시키는 매칭 분자를 만든다(receipt-match.ts:64-116). ③computeAssetLedgerEntry가 (취득가·취득일·매각가)로 감가누계·장부가·처분손익을 산출한다(asset-ledger.ts:71-137, 임대차 가드 포함). ④buildBankJournal이 계정과목 원자를 차/대변 분개로 변환하되 보증금수령=부채·자금이동=GL제외·CMS집금 deposit만 gross+수수료 2분개(item 제외로 이중계상 차단)로 회계적 진실을 인코딩한다(gl-entries.ts:137-168). ⑤simpleVehicleState가 차량 19단계 status를 대표계약과 조합해 '준비축×판매축' 헤드라인 분자('계약됨·차량 소싱 필요')를 만든다. ⑥OperatingSummary가 이 모든 분자를 회사별로 집계한 단일 헤드라인이 된다. [2층·업무흐름 — 분자가 렌터카 생업을 돌린다] 소싱(구매대기)→상품화→계약→인도(운행)→수납→미수→회수(시동제어·내용증명·채권보전)→반납정산→매각. 각 전이는 손입력 스위치가 아니라 원자·분자가 자동 게이팅한다: plate 정규성이 신규차 초기상태를 정하고, prepChecks 완료가 다음 단계로 문을 열며, 미수 분자가 리스크 이슈·회수 SLA를 발화시키고, 반납일이 회차를 면제로 끊어 채권을 확정한다. 응집을 가능케 하는 것은 소수의 '접착제 원자'다. normPlate(공백·O→0·I→1 보정)로 정규화된 차량번호가 차량·계약·보험·과태료·자금을 하나의 물리 축에 꿴다(entity-sync.ts:22-112). companyId/회사코드가 회사격리와 코드발급(assetCode·contractNo)의 뿌리이자 v6 문서ID의 스코프다. 계약번호가 명세↔계약 정확키로 동명이인을 뚫는다. 계정과목이 현금↔회계 변환기다. 그리고 OCR 원본 스냅샷+자연키 dedup+감사로그(before/after)+soft-delete+회사격리가 전체를 감싸는 신뢰 레이어로, 어느 시점이든 재구성 가능하게 한다. 결국 이 ERP는 "데이터를 뜻으로 읽는" 엔진 위에서만 응집하며, 페이지는 분자를 따다 쓰는 얇은 표면이 된다(contract-lifecycle.ts가 명시한 인라인 판단 금지 원칙).
